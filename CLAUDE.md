# b0t

A personal local-AI tool for iOS, with companion styling. On-device, markdown-driven, single user, single device.

The user's b0t lives as a directory of plain markdown files in their Documents folder. A local language model animates those files — Apple Foundation Models by default where the device supports it, or a downloadable open-weight model the user chooses and owns (see ADR-0012). The app is a body for files the user owns.

## Philosophy in one paragraph

b0t is a genuinely useful local-AI tool wearing companion styling — "productivity can be fun." The companion layer is texture over the utility; it must never tax it. Most AI companions are services — you rent a personality, you rent a memory. b0t inverts that. Your b0t is yours. Its identity, memory, and modules are markdown files you can open, edit, share, and version. The model is small, local, and free — and you own the brain too: Apple Foundation Models where the device supports it, or a small downloadable open-weight model you pick. The aesthetic is cassette-futurism — a personal device issued by a fictional small electronics firm circa 1986. The experience is honest about what the system is: a small local model running a heartbeat loop over markdown files. No claims of sentience, no smoke and mirrors.

2. `docs/prd.md` — the what and how. Implementation spec, phased rollout, component-level requirements.

It reframes b0t as tool-first, makes inference engine-agnostic (FM default-when-available / downloadable open-weight otherwise), and contracts v1 to a single non-modular b0t. Before any task touching user-facing copy:

5. `docs/references/voice-and-copy-guide.md` — the voice rules. Apply ruthlessly to every string the user might see.

## Conventions

- **Surface ambiguity, do not silently resolve.** If the PRD and design doc disagree, ask Jamee. If a spec is incomplete, ask. Do not invent.
- **One concern per commit.** Atomic, reversible. Reference the PRD section or ADR in the commit message.
- **Tests are not optional.** Every new public API in `b0tKit` ships with tests. UI views ship with snapshot tests where feasible.
- **Voice and copy guide applies to every string the user might see** — error messages, button labels, settings, tooltips, alerts, App Store metadata. Run every string through `docs/references/voice-and-copy-guide.md`.
- **Privacy is non-negotiable.** No telemetry, no third-party SDKs that phone home, no cloud inference. The one sanctioned network call is **user-initiated model downloads from a pinned, declared source** (ADR-0012); inference itself is always local. Audit any new dependency for network behaviour before inclusion.
- **Aesthetic is non-negotiable.** Cassette-futurism. The display idiom is LCD-forward (backlit monochrome, no bloom/glow) with three semantic highlight colours — yellow `#EAFF3D` (tokens/text), aqua `#3DEAFF` (function/IO), pink `#FF3DEA` (heartbeat/core) — over a muted dark base; the "never blue" rule is overridden (ADR-0016, design §3.5/§9). No glassmorphism, no gradients, no clinical-modern type, no whimsy. The b0t face's visual register (1-bit vs painterly) and the CRT eye-screen treatment are pending Jamee's UI designs (§14 Q1/Q2) — confirm before changing face visuals. See design doc §3.

## Build and test

Two MCP servers are registered in user-level Claude config (`~/.claude.json`):
- **Apple Xcode MCP** (`xcrun mcpbridge`) — `RenderPreview`, `DocumentationSearch`, file ops, Swift REPL.
- **XcodeBuildMCP** — builds, tests, simulators, real devices, LLDB, UI automation.

Project-scoped permissions live in `.claude/settings.json`. Slash commands ship in `.claude/commands/`:

- `/build` — `xcodebuild` build to a generic iOS simulator
- `/test` — `swift test` against the b0tKit package
- `/preview <view>` — Apple MCP `RenderPreview` for a SwiftUI view
- `/fix-build` — build with reasoning; diagnose and propose fixes
- `/implement <feature>` — PRD-driven feature implementation workflow
- `/audit` — App Store submission readiness audit

The Xcode project is generated by `xcodegen` from `project.yml`. After editing `project.yml`, run `xcodegen generate` to refresh `b0t.xcodeproj`. A swift-format pre-commit hook is installed at `.git/hooks/pre-commit` (not committed; `.git/hooks/` lives outside the working tree). It runs `swift format lint --strict` against staged Swift files.

## Definition of done — for any task

1. Code compiles with no warnings.
2. Tests cover the new behaviour and pass.
3. UI changes verified with `RenderPreview`.
4. Voice-and-copy guide followed for any user-facing string.
5. Cassette-futurism aesthetic respected for any visual change.
6. Privacy posture preserved.
7. Performance impact understood.
8. CLAUDE.md or relevant doc updated if the change affects how future tasks should be approached.
9. PR description references the relevant PRD section and any closed open-questions.
